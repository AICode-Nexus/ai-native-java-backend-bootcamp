# 第10课 业务融合与数据助手

## 先看一个真实问题

课程讲到这里，学员通常会问一个非常实际的问题：

“前面这些模型接入、RAG、Tool、Workflow、治理能力，最后到底怎么进入业务？”

这个问题非常关键。  
因为 AI 后端真正的价值，不是单独放一个聊天页，而是进入具体业务系统，帮助用户：

- 查订单
- 看报表
- 问文件
- 做信息整理

但这里同时也是风险最高的区域。  
一旦边界没画清楚，“业务增强”很容易变成“模型直接接管业务执行”。

所以这一课我们故意选一个又典型、又危险、又好教学的场景来展开：  
只读 SQL 数据助手。

## 本课你要拿走什么

- 理解 AI 进入业务系统时应该先做增强层，而不是执行层
- 掌握 Text-to-SQL 场景下最小可控接口长什么样
- 明白为什么“只返回预览、不直接执行”是一种非常强的工程设计

## 为什么这节课要选 SQL 助手来讲

因为它几乎把“AI 与后端业务融合”的核心矛盾全暴露出来了。

一方面，它特别有吸引力：

- 用户可以自然语言提问
- 系统可以自动理解查询意图
- 结果可以直接服务运营、分析、管理决策

另一方面，它又特别危险：

- 生成错误 SQL 会误导业务判断
- 如果放开执行权限，可能伤害生产数据
- 结构不清的输出很难接入后续业务链路

所以用 SQL 助手教学非常合适。  
它能让学员真正理解：

“AI 能做什么”和“AI 应该做到哪一步”，不是同一个问题。

## 当前 demo 的接口边界是怎么画的

这节课最值得带着学员读的，就是 `SqlAssistantController`。

当前接口的处理链路非常克制：

1. 接收业务问题
2. 校验问题是否包含高风险写操作意图
3. 生成只读 SQL 预览
4. 返回结构化响应

注意，这里没有做的一件事同样重要：

“它不直接执行 SQL。”

这意味着系统当前只承担：

- 理解问题
- 生成建议
- 提供预览
- 标记风险

而不承担：

- 直接落库执行
- 修改生产数据
- 自动触发业务写操作

这条边界，是这一课必须讲透的核心。

## 为什么“只返回预览”是非常强的设计

很多初学者会觉得，既然已经能生成 SQL，为什么不顺手执行掉？  
但真正成熟的后端设计，恰恰知道什么时候该停下来。

只返回预览有几个明显好处：

- 允许人工审核
- 允许后续接审批流
- 允许前端先展示再确认
- 允许业务系统保留最终执行权

更重要的是，它把 AI 从“执行主体”降回“智能增强层”。  
这会让整个系统更容易上线，也更容易和现有业务系统融合。

## 为什么结构化响应比“生成一段 SQL 文本”更重要

如果这个接口只返回一段字符串，前端和业务系统几乎没法稳定接入。  
当前 demo 返回的是一个结构化对象，至少包含：

- `sqlPreview`
- `safe`
- `summary`

这几个字段分别承载了不同职责：

- `sqlPreview` 让用户看到系统建议
- `safe` 让调用方知道当前输出是否满足只读边界
- `summary` 让接口结果更易于展示和解释

这也正是 AI 后端和普通聊天式应用的重要区别之一：  
真正进入业务系统时，结构化结果远比“看起来很聪明的自然语言”更有价值。

## 为什么输入也要做约束，而不是只约束输出

`SqlAssistantController` 中有一个很值得讲的点：  
系统不是等 SQL 生成出来之后再看风险，而是在用户问题进入时就先做了只读意图约束。

这说明安全边界最好是多层的：

- 输入层约束问题意图
- 输出层约束 SQL 语义
- 业务层保留最终执行权

这样即使某一层不够完美，系统仍然有机会在其他层把风险拦住。  
这正是工程化防线和“单点信任模型”之间的差别。

## 这一课如何和前面能力串起来

这节课不要孤立讲。  
它其实是前面所有能力第一次向业务系统汇合的地方。

可以这样串：

- 第 1 到 3 课解决模型接入和结构化表达
- 第 4 到 6 课解决知识与检索增强
- 第 7 课解决工具与权限边界
- 第 8 课解决长链路任务和异步处理
- 第 9 课解决治理与回归
- 第 10 课开始把这些能力真正嵌进业务接口

学员一旦看懂这条主线，就会意识到：  
所谓“AI + Java 后端”，不是多加一个聊天接口，而是把 AI 能力以受控方式融入原有业务架构。

## 课堂建议怎么讲

### 第一步：先讲业务侧真正想要的不是“模型”，而是“提效”

让学员站在业务角度理解需求：

- 运营想更快查数
- 销售想更快看趋势
- 管理层想更快拿到摘要

AI 在这里不是目的，而是接口增强方式。

### 第二步：再讲为什么要克制

重点强调：

- 可以生成 SQL
- 但不直接执行 SQL
- 可以给建议
- 但不直接接管业务动作

这会帮助学员建立非常宝贵的工程判断力。

### 第三步：最后对着测试讲边界

`SqlAssistantControllerTest` 里两条用例很适合课堂讲解：

- 正常问题要返回只读 SQL 预览
- 高风险问题要直接拒绝

这两条测试加起来，已经足够表达“业务融合不是无限放权”。

## 学员最容易踩的坑

- 误区一：觉得能生成 SQL 就应该直接执行
- 误区二：只返回自然语言说明，不返回结构化字段
- 误区三：把 AI 直接塞进业务核心写操作路径
- 误区四：把“业务增强”误理解成“模型接管业务系统”

## 本课小结

这一课告诉学员，AI 后端真正开始产生业务价值的时刻，是它进入具体业务接口的时候。  
但越接近业务核心，越要克制地设计边界。

先做增强层，先做预览层，先做只读层，这是训练营里非常值得学员带走的一种落地范式。
