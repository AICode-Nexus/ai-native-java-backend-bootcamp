# 第11课 全链路整合与生产化

## 先看一个真实问题

课程最后一课，学员往往会回到一个最现实的问题：

“现在这套 demo 已经能聊天、能检索、能上传文档、能做 RAG、能做工具调用、也有治理层了。那它离生产到底还差什么？”

这个问题之所以重要，是因为很多训练营项目都会卡在这里：

- 单个功能看起来都能演示
- 但整条链没有真正收束
- 学员知道每节课讲了什么
- 却说不清楚如何把这些能力拼成一个可落地系统

所以这一课的任务，不是再补一个新功能，而是把前面 11 课学到的东西真正连成一张系统地图，并且告诉学员：

“从训练营 demo 到真实生产，不是推倒重来，而是分层替换、逐步演进。”

## 本课你要拿走什么

- 把整套 AI-Native Java 后端主链路串成一个完整故事
- 理解从 demo 到生产应优先补哪些层面
- 掌握一条更现实的落地路线，而不是空泛谈架构升级

## 到这一课为止，我们到底完成了一条什么主链路

如果只从目录看，仓库会显得模块很多；  
但如果从系统角度看，整套课程其实是在逐步搭一条非常明确的主链路。

可以把它概括成下面这张图：

1. 用户请求进入 Spring Boot API
2. 系统决定是走聊天、RAG、工具还是业务增强接口
3. 如有需要，系统先检索知识、读取缓存或调用工具
4. 模型基于受控上下文生成结果
5. 结果以结构化形式返回
6. 日志、指标、评测、CI 负责持续治理

如果学员能把这条链顺着讲出来，说明这门课最关键的目标已经达到了。

## 为什么说这不是“新框架”，而是“新系统组织方式”

这一课一定要帮助学员跳出一个误区：  
AI-Native Java 后端不是要替换 Spring Boot，也不是要抛弃传统工程体系。

真正变化的是系统中心从哪里开始组织。

### 过去的中心通常是事务和表

很多传统后端项目的核心路径是：

- 接请求
- 验参数
- 查表或改表
- 返回结果

### 现在的中心会多出推理、检索、工具和治理

AI 后端的核心路径则可能变成：

- 接请求
- 识别意图
- 检索上下文
- 调模型或工具
- 结构化输出
- 记录风险和质量状态

Spring Boot 还是骨架，Java 还是主工程语言，  
但系统设计重点已经从“事务流”扩展到“推理流 + 检索流 + 治理流”。

## 从 demo 走向生产，优先要补哪几层

这是这一课最需要给学员讲清楚的部分。  
生产化并不是一次性做“大重构”，而是按层替换。

### 第一层：把主链路先稳定住

你要先确认：

- 聊天主链路是否可用
- RAG 主链路是否可用
- 上传与知识入库是否可用
- 数据助手这类业务增强接口是否可用

如果主链路本身还不稳，谈生产化只会变成空谈。

### 第二层：把底层 stub 替换成真实基础设施

当前 demo 为了教学刻意用了很多最小实现。  
真实落地时常见的替换方向包括：

- stub model -> 真实模型供应商
- 内存缓存 -> Redis
- 内存检索或占位实现 -> Elasticsearch / 向量库
- 内存任务发布消费 -> MQ / workflow engine

这类替换如果前面边界设计做得对，应该是局部演进，而不是推倒重做。

### 第三层：把治理补成上线门槛

生产化一定要把这些能力补齐：

- 安全策略
- 请求日志
- 模型与缓存指标
- 关键场景 evals
- 最小 CI 检查

如果没有这层，系统上线后即使能跑，也很难长期维护。

### 第四层：再考虑复杂度升级

只有前面三层站稳了，才值得继续讨论：

- 多模型路由
- 多租户隔离
- 多环境配置
- 更复杂的 Agent / Workflow 编排

这也是课程最后要帮学员建立的一种优先级判断力。

## 把课程主线压缩成三个系统故事

如果你要带学员复盘整套训练营，很适合把内容收束成三个故事。

### 故事一：系统如何接入 AI

这一条主线覆盖前 1 到 3 课：

- Spring Boot 如何接模型
- 对话和流式接口如何设计
- Prompt、上下文、结构化输出如何受控

这条线回答的是：  
“模型调用如何成为后端能力，而不是零散脚本。”

### 故事二：系统如何拥有企业知识和工具能力

这一条主线覆盖前 4 到 8 课：

- 文档怎么进入系统
- RAG 如何回答企业知识
- Redis / ES / 向量检索如何协同
- Tool 和 Workflow 如何把能力扩展出去

这条线回答的是：  
“AI 后端为什么不只是一个聊天接口。”

### 故事三：系统如何进入真实业务和真实环境

这一条主线覆盖前 9 到 11 课：

- 治理如何形成最小闭环
- 数据助手这类业务接口如何克制融合
- demo 如何逐步替换成生产实现

这条线回答的是：  
“为什么真正落地靠的是工程体系，而不是 prompt 小技巧。”

## 课堂最后可以怎么收束

### 第一种收束方式：让学员自己画系统图

让学员画出：

- API 层
- AI Service 层
- Retrieval / Tool / Workflow 层
- Data / Infra 层
- Governance 层

只要他能画清楚这张图，说明整套课程已经真正进入理解层。

### 第二种收束方式：让学员回答三个问题

1. 我的 AI 后端主链路是什么  
2. 我的风险边界和治理点在哪里  
3. 如果明天要接入真实业务，我优先替换哪几层  

这三个问题，几乎能检验学员是否真的把课程带走了。

## 学员最容易踩的坑

- 误区一：以为生产化就是把 demo 全部重写一遍
- 误区二：功能越多越接近落地，忽略主链路稳定性
- 误区三：先追多模型、多 Agent、多环境，跳过治理和评测
- 误区四：学完每一课都懂一点，但说不清整套系统怎么串起来

## 全课小结

这门训练营真正想交给学员的，不是某个具体框架的 API 细节，而是一种新的后端组织能力：

- 会把模型接成工程能力
- 会把知识、工具、工作流接进系统
- 会用治理把 AI 风险拉回可控范围
- 会把 demo 沿着清晰路径演进到生产

如果学员结课时已经能用 Java 和 Spring Boot 把这套主线讲清楚、搭出来、再逐步替换底层实现，那么这门课就真正完成了它的目标。
