# AI-Native Java Backend Bootcamp 课程网站设计文档

## 背景

当前仓库 `ai-native-java-backend-bootcamp` 已经具备一套完整的课程内容资产：

- 顶层 `README.md` 负责课程总览
- 12 节课分别存放在仓库根目录
- `课后练习/` 承载对应练习
- `demo/` 承载 Java / Spring Boot 主项目

但它目前仍然是“课程资料仓库”，还不是一个真正的课程网站。  
与之对应，参考仓库 `ai-native-frontend-bootcamp` 已经形成了一套独立的课程站范式：

- 独立前端应用负责站点壳层
- 首页与 `/learn` 学习区分工明确
- 头部、侧边栏、搜索、Markdown 课程页、移动端抽屉形成统一交互
- 仓库根目录课程稿作为站点内容来源

用户希望当前后端训练营仓库也具备同样的课程网站能力，并且与参考仓库保持同一产品家族感：风格一致、交互一致、结构一致，但内容语义应符合 Java 后端训练营，而不是直接换皮。

## 用户已确认的关键决策

### 1. 站点放置方式

采用独立前端站点目录：

- 新增 `website/`
- 保留现有 `demo/` 作为 Java 主项目
- 不把课程网站塞进现有 `demo/`

### 2. 功能范围

一期范围为：

- 课程网站首页
- `/learn` 课程总览页
- `/learn/[lessonId]` 单课学习页
- 与参考站一致的导航、搜索、移动端交互和 Markdown 阅读体验

### 3. 一致性目标

采用“参考站基础上的后端版适配”：

- 保持页面结构和核心交互一致
- 保持同一产品系列感
- 允许做少量更符合 Java 后端主题的视觉和内容语义调整

### 4. 内容修订边界

允许修与站点展示直接相关的内容问题，包括：

- 课程副标题、标签、摘要、时长等站点元数据
- 课程标题与术语口径统一
- 明显错漏、表达跳跃和展示密度问题

不在本轮进行：

- 12 节课的大规模重写
- 脱离课程网站目标的教学结构重做
- 新课程主题扩展

## 目标

新增一个独立的课程网站，使本仓库同时具备以下能力：

1. 作为公开入口，清晰展示 AI-Native Java Backend Bootcamp 的定位、课程结构和主线价值
2. 作为学习入口，提供和参考站一致的课程浏览、搜索和阅读体验
3. 作为课程壳层，直接消费仓库根目录课程 Markdown，避免内容双份维护
4. 作为系列站点，与 `ai-native-frontend-bootcamp` 在产品结构和交互语言上保持连续性
5. 在不破坏现有 Java `demo/` 的前提下，为后续部署与持续演进留出空间

## 非目标

以下内容不属于本次设计的一期目标：

1. 不把课程网站与 Spring Boot 运行时做强耦合整合
2. 不在一期内实现复杂账号体系、评论系统或后台 CMS
3. 不把所有课程内容重构为 MDX 或数据库存储
4. 不把网站目标扩展成完整营销官网或社区平台
5. 不改造现有 `demo/` 的 Java 业务能力

## 总体方案

推荐采用“独立课程站 + 根目录课程稿 + 并列 Java Demo”的结构：

```text
.
├── website/                  # 新增课程网站（Next.js）
├── demo/                     # 现有 Java / Spring Boot 主项目
├── docs/
│   └── superpowers/
│       ├── specs/
│       └── plans/
├── 第0课-认知重构/
├── 第1课-Spring-Boot-接入模型/
...
└── 第11课-全链路整合与生产化/
```

其中：

- `website/` 负责站点 UI、路由、搜索、课程装配和阅读体验
- 仓库根目录各课 `final-content.md` 继续作为课程正文唯一来源
- `demo/` 保持 Java 教学主项目定位，不因课程站引入前端站点壳层耦合

## 技术方向

为保证与参考仓库的实现和交互尽量对齐，课程站技术方向建议直接沿用其主栈：

- `Next.js 15`
- `React 19`
- `TypeScript`
- `Tailwind CSS v4`
- 必要的轻量客户端交互组件
- 与参考站相同思路的 Markdown 渲染、代码高亮和 Mermaid 支持

这里的关键不是“为了炫技选前端栈”，而是为了保证系列一致性和复用同类站点结构：

- 页面壳层模式一致
- 组件职责一致
- 学习区体验一致
- 后续维护心智一致

## 信息架构

### 站点级路由

一期建议只做三类核心路由：

1. `/`
   课程门户首页
2. `/learn`
   课程总览页
3. `/learn/[lessonId]`
   单课学习页

### 首页 `/`

首页承担公开入口和仓库导航职责，建议包含以下区块：

1. Hero 区
   展示课程标题、定位、受众、课时、主技术栈和进入学习区入口
2. 后端技术全景图
   用“系统层次”而非“前端工具层次”组织内容
3. 课程总览预览
   展示 12 节课的主线
4. 核心原则
   强化课程方法论
5. 资源入口
   链接到仓库 README、Java Demo、社区页等

### 学习区 `/learn`

学习区总览页延续参考站的阅读节奏，但内容改为后端训练营语义：

- 顶部课程定位摘要
- 课程关键数据，如课时、受众、学习目标
- 技术全景图
- 12 节课课程列表
- 核心原则区块

### 单课页 `/learn/[lessonId]`

单课页以阅读为中心，支持：

- 课程头信息
- 标签与时长
- Markdown 正文
- 右侧目录跟随
- 上一课 / 下一课切换
- 移动端友好的导航和内容布局

## 页面与交互设计

以下交互应尽量与参考站保持一致：

### 顶部 Header

- 固定在顶部
- 左侧为站点识别和回到 `/learn` 的入口
- 右侧为搜索入口和站点导航
- 移动端提供汉堡按钮控制侧边导航

### 左侧 Sidebar

- 桌面端为常驻课程导航
- 移动端为抽屉式导航
- 支持当前课程高亮
- 顶部有“课程总览”入口

### 搜索 Dialog

- 支持快捷键打开
- 支持按标题、副标题、标签、正文片段搜索
- 支持键盘上下选择和 Enter 打开
- 无结果时提供清晰空状态

### Markdown 阅读体验

- 支持代码高亮
- 支持表格、引用块、列表、链接
- 支持 Mermaid
- 支持自动生成目录
- 保持长文阅读友好，不引入花哨但不稳定的动态效果

### 上下课导航

- 单课页底部提供上一课 / 下一课入口
- 强化课程主线连续性

## 视觉方向

整体视觉策略为：

**保持参考站的产品节奏与信息密度，但把视觉语义从“前端工具链”切换为“后端系统链路”。**

具体要求：

1. 保持简洁、克制、内容优先的整体调性
2. 保持课程站系列一致性，不做完全不同的品牌语言
3. 主色可以保留电蓝系家族感，但应降低前端产品气质中过强的“组件感”
4. 首页的“技术全景图”与标签系统应强调层次、边界、链路、治理
5. 文案与视觉导视更多使用“接入、检索、编排、治理、生产化”这样的后端语义

## 内容装配模型

课程内容建议拆成两层：

### 1. 正文层

继续使用仓库根目录各课的 `final-content.md` 作为正文唯一来源。

### 2. 元数据层

在 `website/src/lib/lessons.ts` 维护站点级元数据，例如：

- `id`
- `number`
- `title`
- `subtitle`
- `duration`
- `tags`
- `dirName`
- `summary`

这样做的原因是：

1. 避免正文和站点展示文案双份维护
2. 为首页、课程总览、搜索结果提供更适合导航消费的短内容
3. 允许对展示层信息进行精修，而不强迫每节课正文都承载导航元信息

## 内容质量修订策略

基于当前课程稿的快速检查，可以得出两个结论：

1. 整体课程结构已经比较整齐，多数课程都具备统一的二级标题骨架
2. 站点化之后，最需要补强的是元数据统一性，以及少数课程在开头信息密度上的展示友好度

因此本轮内容修订建议采用“轻修订、强统一”的策略：

### 可做的修订

1. 统一课程标题口径
2. 补齐每节课的 `subtitle`
3. 补齐每节课的 `duration`
4. 补齐每节课的 `tags`
5. 为每节课提炼 1 到 2 句 `summary`
6. 修明显错别字、术语不统一和链接问题
7. 对少数课程开头做轻量整理，使首页和搜索展示更连贯

### 暂不做的修订

1. 不重写全部课程正文
2. 不改变 12 节课的主线排序
3. 不把课程稿从 Markdown 全量迁移到别的内容系统
4. 不为了站点效果引入大量营销式文案

## 目录与模块设计

`website/` 内建议对齐参考站的组织方式：

```text
website/
├── src/app/
│   ├── page.tsx
│   ├── learn/page.tsx
│   ├── learn/[lessonId]/page.tsx
│   └── globals.css
├── src/components/learn/
│   ├── header.tsx
│   ├── sidebar.tsx
│   ├── search-dialog.tsx
│   ├── lesson-content.tsx
│   └── mobile-nav-context.tsx
├── src/lib/
│   ├── lessons.ts
│   ├── site-content.ts
│   ├── lesson-search.ts
│   ├── lesson-search.server.ts
│   └── lesson-markdown.ts
└── public/
```

模块职责如下：

- `page.tsx`
  首页门户
- `learn/page.tsx`
  课程总览
- `learn/[lessonId]/page.tsx`
  单课装配页
- `header.tsx`
  站点顶部导航
- `sidebar.tsx`
  课程导航
- `search-dialog.tsx`
  搜索入口与弹层
- `lesson-content.tsx`
  Markdown 渲染与目录体验
- `lessons.ts`
  课程元数据
- `site-content.ts`
  首页和总览页的站点级内容配置
- `lesson-search.*`
  搜索数据与匹配逻辑
- `lesson-markdown.ts`
  正文预处理和目录生成

## 错误处理与构建约束

为避免课程站上线后出现静默错误，建议明确以下规则：

1. 课程元数据和目录映射必须一致
2. 某节课如果缺少 `final-content.md`，构建应显式失败或给出明确开发提示
3. Mermaid 渲染失败时不应导致整页不可读
4. 搜索无结果时应提供稳定空状态
5. 课程总览页必须能完整列出 12 节课

## 测试策略

一期至少覆盖以下测试：

1. 课程元数据与目录映射一致性测试
2. `lesson-markdown.ts` 的标题清洗和目录生成测试
3. `lesson-search.ts` 的标题、标签、正文片段命中测试
4. `lesson-content.tsx` 的关键 Markdown 渲染测试
5. 页面级最小验证：
   - `/learn` 能列出全部课程
   - 单课页能正确读取正文

## 验收标准

本设计完成后的验收标准如下：

1. 仓库新增独立 `website/` 课程站，不影响现有 `demo/`
2. 首页、`/learn` 和单课页形成清晰稳定的信息架构
3. 头部、侧边栏、移动端抽屉、搜索、目录、上下课切换等核心交互与参考站保持一致的使用体验
4. 课程正文直接读取仓库根目录各课 `final-content.md`
5. 课程元数据补齐并统一，站点展示不依赖临时硬编码文案拼接
6. 对展示质量直接相关的课程内容问题完成有边界的修订
7. 至少具备基础测试覆盖课程映射、搜索和 Markdown 处理

## 风险与控制

### 风险 1：网站与课程正文分叉

控制方式：

- 正文继续只保留在根目录 Markdown
- 网站只维护元数据和装配逻辑

### 风险 2：为了“像参考站”而丢失后端课程自身语义

控制方式：

- 复用结构和交互，不机械复刻文案
- 首页技术全景、标签和导视改为后端版表达

### 风险 3：内容修订范围失控

控制方式：

- 只修与网站展示直接相关的问题
- 不做整仓重写

## 实施顺序建议

后续进入 implementation plan 时，建议按以下顺序推进：

1. 搭建 `website/` 基础工程
2. 建立课程元数据和目录映射
3. 打通 `/learn` 与单课正文读取
4. 复用并适配 Header / Sidebar / Search / LessonContent
5. 完成首页后端版内容适配
6. 补齐课程元数据并做轻量内容修订
7. 补测试并做最终验收

## 结论

本次设计的核心判断是：

**要实现与 `ai-native-frontend-bootcamp` 一致的课程网站风格与交互，最合理的方式不是改造现有 Java `demo/`，而是在当前仓库中新增一个独立的 `website/` Next.js 课程站，并让它直接消费仓库根目录课程稿。**

这种方式能够同时满足：

- 系列一致性
- 展示完整性
- 内容单一来源
- 对现有 Java 教学主项目的最小扰动
- 后续可持续演进
