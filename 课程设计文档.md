# 《AI-Native Java Backend Bootcamp》课程设计文档

## 一、课程要解决什么问题

这套课程的核心命题不是“Java 如何调用一个模型接口”，而是：

**如何用 Java / Spring Boot 构建一套可上线、可治理、可持续演进的 AI-Native 后端系统。**

很多团队已经能快速做出 AI Demo，但真正进入业务后会立刻遇到新的问题：

- 模型接入分散，边界不清
- 对话、RAG、Tool、Workflow 混在一起
- 功能能演示，但安全、日志、评测和成本治理缺失
- 业务接口看起来很智能，但没有真正可上线的系统组织方式

这套训练营的目标，就是把这些问题压缩成一条清晰的 Java 后端主线。

## 二、课程定位

### 1. 主定位

面向进阶开发者，讲清 AI 与 Java 后端主流工程体系如何深度结合，重点不在“AI 会什么”，而在“AI 能如何变成后端系统能力”。

### 2. 主受众

- AI 应用开发者
- 需要补足后端体系能力的前端或全栈工程师
- 已经会基本 Java / Spring Boot，希望进入 AI 后端方向的工程师

### 3. 次要对照受众

课程主线是 Java，但会保持对 Go、Python、TypeScript 的工程对照意识。  
这些语言不是主讲对象，而是帮助学员理解 Java 方案为什么适合企业主后端。

### 4. 课程输出

学员完成整套课程后，应该具备三类能力：

- 能搭建一条 AI 后端主链路
- 能说清系统边界与风险治理点
- 能把 Demo 沿着清晰路径演进到生产

## 三、教学主线

### 1. 主业务场景

整套课程围绕“企业知识助手”展开。  
这是一个非常适合教学的业务主线，因为它天然能串起：

- 模型接入
- 对话接口
- 文件上传与知识入库
- RAG 检索增强
- Tool Calling
- Workflow 与异步任务
- 工程治理
- 业务融合与生产化

### 2. 为什么不用“散点式 Demo”

如果每一节课都只是做一个小功能，学员最终会记住很多名词，却很难建立系统观。  
企业知识助手的价值在于它能让每节课都成为同一套系统的增量，而不是无关示例。

### 3. 主线系统地图

课程最终希望学员建立下面这张系统地图：

- API 层
- AI Service 层
- Retrieval / Tool / Workflow 层
- Data / Infra 层
- Governance 层

这张地图是整套课程的隐形主轴。  
每一课新增的代码、文档和练习，都应该能挂回这张地图上。

## 四、教学原则

### 1. 业务链路优先

课程按业务能力组织，不按中间件清单组织。  
先讲“问题是什么、系统怎么接”，再讲“底层基础设施如何承载”。

### 2. 深讲 Java，横向对照其他语言

主线始终落在 Java / Spring Boot。  
Go、Python、TypeScript 只承担对照和补充角色，不抢主叙事。

### 3. 工程边界优先于炫技效果

课程强调：

- 模型接入边界
- 检索边界
- Tool 与 Agent 边界
- Workflow 与同步接口边界
- 治理边界

避免把课程带成“AI 能做很多事”的功能秀。

### 4. 治理从中段开始持续引入

安全、日志、指标、评测、CI 不是收尾附录，而是课程中后段的重要能力线。

### 5. 当前实现与目标演进分开讲清

训练营当前仓库为了教学效率，采用了大量最小可教实现。  
文档中必须明确区分：

- 当前仓库已实现什么
- 后续生产化可替换成什么

避免学员误以为“文档写到的技术栈都已经在 demo 中完整落地”。

## 五、当前仓库的教学实现与目标技术栈

## 5.1 当前仓库真实教学实现

当前 `demo/` 的重点是把主链路和边界教清楚，已经具备的能力包括：

- Java 21 + Spring Boot 3.x 最小工程骨架
- 健康检查与统一响应结构
- 同步聊天与流式聊天接口
- 文档上传、切片与内存入库
- 最小 RAG 检索与引用返回
- Redis / ES / 向量检索的边界抽象与协同骨架
- Tool Registry、Workflow、异步任务骨架
- 安全策略、请求日志、指标门面、最小 evals、CI
- 只读 SQL 预览型业务融合案例

## 5.2 后续生产化目标技术栈

当课程从教学 Demo 进入真实业务演进时，目标技术栈可以逐步替换为：

- 商业或开源大模型供应商
- Redis
- Elasticsearch
- 向量数据库
- RabbitMQ 或其他 MQ / workflow engine
- 更完整的模型网关、日志、监控、评测体系

这里的关键不是具体产品名，而是：  
边界先稳定，底层再替换。

## 七、单课教学建议

建议每节课按同一教学节奏推进，这样学员更容易建立稳定认知：

1. 先用一个真实问题开场
2. 再说明本课要拿走什么
3. 再落到当前仓库代码位置
4. 再拆核心概念与边界
5. 最后讲课堂 demo、易踩坑和课后练习

这一节奏已经在当前 12 篇 `final-content.md` 中统一下来，讲师可以直接沿用。

## 八、课程资产设计

当前仓库的课程资产已经按三层组织：

### 1. 正文层

- 每课一篇 `final-content.md`
- 统一采用“真实问题 -> 收获 -> 代码映射 -> 概念拆解 -> 课堂讲法 -> 易踩坑 -> 小结”的结构

### 2. 代码层

- `demo/` 作为贯穿主项目
- 代码优先体现边界和最小主链路
- 单元测试和接口测试作为教学解释入口

### 3. 练习层

- `课后练习/` 每课一份练习文档
- 每份练习都分为基础题、进阶题、挑战题
- 每份练习都包含参考位置、建议路径、交付物和验收标准

## 九、评估与验收方式

这套训练营不建议只用“做完功能”作为验收标准。  
更合理的验收应该同时看三件事：

### 1. 是否跑通主链路

例如：

- 模型调用是否通
- RAG 是否返回引用
- 工具是否受控
- Workflow 是否显式暴露失败步骤

### 2. 是否说清边界

例如：

- 为什么 SQL 助手默认只做预览
- 为什么检索为空必须降级
- 为什么 Tool 和 Agent 不能混写

### 3. 是否具备最小工程治理意识

例如：

- 是否补了安全规则
- 是否有最小日志或指标
- 是否有回归样例

## 十、讲师使用建议

### 1. 不要过早展开中间件细节

课程前半段要优先让学员理解边界和链路，不要一开始就陷进基础设施安装和运维细节。

### 2. 多从测试反推设计意图

当前仓库里很多测试都很适合直接拿来讲：

- `ChatControllerTest`
- `TextChunkerTest`
- `RagServiceTest`
- `ToolRegistryTest`
- `KnowledgeWorkflowServiceTest`
- `SqlAssistantControllerTest`

这些测试天然能帮助学员理解“系统为什么这样设计”。

### 3. 课堂上持续把内容挂回同一张系统地图

每一课都建议反复提醒：

- 这一课新增了哪一层能力
- 它与前一课是什么关系
- 它在企业知识助手主链路里承担什么角色

## 十一、课程完成标志

如果学员在结课时能清楚回答下面三个问题，这套课程就算真正达成目标：

1. 我的 AI 后端主链路是什么  
2. 我的边界和风险点在哪里  
3. 我如何把当前 demo 逐步迁移到真实业务系统  

当学员能把这三个问题说清楚，并能在 Java / Spring Boot 中落成最小实现时，这套训练营就不再只是“学过一些 AI 概念”，而是真正建立了 AI-Native 后端能力。
