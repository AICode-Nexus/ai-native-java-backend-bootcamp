# 第4课 练习

## 练习定位

这一课练的是“让 AI 参与写业务代码，但不让它悄悄把边界写坏”。重点不是让 AI 多写，而是练习你如何提前声明职责、事后检查分层，并用测试守住行为。

## 基础题

### 任务

选择一个 Service 方法，先写清职责说明，再让 AI 生成一版实现或伪实现。

### 职责说明至少包含

- 这个方法负责什么
- 不负责什么
- 它依赖哪些输入
- 它应该返回什么
- 哪些异常或边界要显式处理

### 建议输入模板

```md
目标方法：
职责边界：
禁止出现的职责：
输入输出：
依赖对象：
必须保留的业务规则：
```

## 进阶题

### 任务

人工检查 AI 生成结果的分层边界，并列出至少 3 个必须修改点。

### 必查项

- Controller 是否塞进了业务判断、异常拼装或底层细节
- Service 是否直接承担了它不该知道的 HTTP 语义
- 是否出现跨层直接访问、硬编码配置、临时 Map 乱传值等问题
- 是否为了“看起来完整”而引入了无必要的抽象类或工具类

### 输出格式建议

```md
问题：
风险：
修正动作：
如果不改，后续会在哪一步出问题：
```

## 挑战题

### 任务

补一条测试，证明你完成边界修正后，关键行为仍然正确。

### 建议做法

- 如果改的是控制器边界，参考 `ChatControllerTest`
- 如果改的是业务逻辑边界，参考 `ChatServiceTest`
- 如果你还没真的改代码，也至少写出应补的测试场景、预期断言和失败信号

### 反例提醒

“代码能跑” 不等于边界正确。只要 Controller 开始拼业务规则，或者 Service 开始处理展示层细节，后续重构成本就会迅速升高。

## 建议交付物

- 一份方法职责说明
- 一份 AI 生成结果的边界审查记录
- 一条新增测试或测试设计说明

## 验收要点

- 你明确写出了“该方法不负责什么”
- 至少找出了 3 个边界问题，而不是只说“代码可以更优雅”
- 测试验证的是业务行为或接口行为，不是实现细节
- 你的修正建议能回到真实文件和分层结构，而不是停留在抽象原则
