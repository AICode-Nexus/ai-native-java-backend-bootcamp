# 第0课 认知重构

## 先看一个真实问题

很多团队说自己在做 AI 后端，实际做法只是给原来的 Java 服务多加了一个“调用大模型”的接口。短期看，这样确实能跑起来；但只要业务继续推进，很快就会遇到一串连锁问题：

- 聊天接口和原有业务接口混在一起，分层越来越乱
- 文档问答、Tool Calling、Workflow 没有稳定边界
- 模型结果能展示，但不能追溯、不能评测、不能治理
- 系统看起来“很智能”，实际不敢上线

所以这门训练营第一课不急着写代码，而是先把认知框架摆正。我们要教的不是“Java 怎么调一个模型”，而是“如何用 Java / Spring Boot 组织一套可持续演进的 AI-Native 后端系统”。

## 本课你要拿走什么

学完这一课，学员应该能清楚回答三件事：

1. AI-Native 后端和传统 CRUD 后端的核心差别是什么
2. Chat、RAG、Tool、Workflow、Governance 在系统里分别放哪一层
3. 为什么本训练营必须围绕一条业务主线做，而不是做十几个散点 Demo

## 这门课在整条主线里解决什么问题

本训练营的主线业务是“企业知识助手”。它不是一个随便挑的例子，而是一个天然能够串起整套能力链的典型场景：

- 用户先发起自然语言问题
- 系统需要调用模型
- 回答经常依赖企业知识库
- 复杂请求需要工具调用和异步工作流
- 真正上线时还必须考虑安全、评测、日志、成本和交付

也就是说，只要主线业务选对了，后面每一课都不是孤立技术点，而是在给同一个系统补一层能力。

## 传统 CRUD 后端和 AI-Native 后端到底差在哪

### 1. 请求输入变了

传统后端的输入大多是结构化字段，比如订单号、用户 ID、状态枚举。  
AI-Native 后端的输入会变成：

- 自然语言
- 文档内容
- 检索结果
- 工具调用意图
- 多轮会话上下文

这意味着系统不再只是在做“固定字段到固定 SQL”的映射，而是在做“开放输入到受控推理”的组织。

### 2. 系统中间层变了

传统后端常见分层是：

`Controller -> Service -> Repository -> DB`

AI-Native 后端更合理的分层会变成：

`API 层 -> AI Service 层 -> Retrieval / Tool / Workflow 层 -> Data / Infra 层 -> Governance 层`

这里最关键的变化，是系统里多了几类传统 CRUD 很少正面处理的能力：

- 模型接入与结果约束
- 检索增强与引用链路
- 受控工具调用
- 长链路异步流程
- 安全、评测、日志与指标

### 3. 成功标准变了

传统后端常见的成功标准是“接口通了、事务正确、性能稳定”。  
AI-Native 后端除了这些，还要多回答几类问题：

- 回答是否 grounded
- 工具调用是否越权
- 结果是否可解释
- 模型成本是否可控
- 回归样例是否稳定

## 这门课要先建立哪张系统地图

建议课堂上先给学员画出一张“能力地图”，哪怕是手写白板版也可以。至少要包含下面五层：

### API 层

面向外部用户和前端的入口，例如：

- `/api/chat`
- `/api/knowledge/ask`
- `/api/documents/upload`
- `/api/data-assistant/sql`

### AI Service 层

负责把业务请求翻译成模型可理解、可控制的调用形式，例如：

- 模型网关
- Prompt 参数化
- 输出解析

### Retrieval / Tool / Workflow 层

这是 AI 后端最核心、也最容易混乱的一层。它包含：

- 检索召回
- 工具注册和执行
- 固定流程编排

### Data / Infra 层

包括：

- MySQL
- Redis
- Elasticsearch
- 向量检索底座
- MQ

### Governance 层

包括：

- 安全策略
- 请求日志
- 指标门面
- 评测样例
- CI

## 为什么这一课要先讲“边界”

很多 AI 项目越做越乱，不是因为技术选型错了，而是因为边界从一开始就没讲清。  
比如：

- 把检索逻辑直接塞进 Controller
- 把 Tool 调用和 Workflow 写进一个万能 Service
- 把安全策略当成前端提示文案
- 把 RAG 当成“先查一下再回答”

所以第一课的任务不是让学员记术语，而是让大家先会画边界图。只要边界清楚，后面每一课的新增代码都能自然挂到地图上。

## 推荐课堂讲法

### 开场

先用一句话打破预期：

“这套课不是教你怎么从 Java 调一个模型，而是教你怎么把 AI 变成后端系统能力。”

### 中段

把传统 CRUD 后端和 AI-Native 后端的系统图并排讲。  
讲的时候不要先堆技术名词，而是从用户请求的变化开始讲。

### 收尾

落回当前仓库，让学员知道后面每一课不是独立主题，而是在给同一个企业知识助手逐层加能力。

## 学员最容易踩的坑

- 误区一：把 AI 后端理解成“多一个 `/chat` 接口”
- 误区二：先接一堆基础设施，再回头找主线业务
- 误区三：把 Agent 当成能吞掉所有逻辑的万能层
- 误区四：把治理当成最后才补的锦上添花

## 本课小结

第一课真正要完成的不是功能，而是认知重构。  
学员如果能理解“AI-Native 后端是一套新的系统组织方式，而不是传统后端加一个模型 SDK”，后面每一课都会更顺。下一课开始，我们正式从模型接入边界下手，把系统的第一块工程地基搭起来。
