# AI-Native Java Backend Bootcamp 演讲大纲

## 总体讲法

整套课程建议按三个系统故事来讲，而不是按一组零散技术名词来讲：

1. 系统如何接入 AI
2. 系统如何拥有知识、工具和流程能力
3. 系统如何进入真实业务并走向生产

每一课建议都遵循同一节奏：

1. 用真实问题开场
2. 讲本课要拿走什么
3. 对照仓库代码解释边界
4. 强调一到两个最重要的风险点
5. 用练习把课堂内容变成可执行任务

## 第0课 认知重构

### 开场问题

- 为什么很多“AI 后端项目”其实只是原有系统多了一个模型调用接口

### 本课必须讲清楚

- AI-Native 后端不是传统 CRUD 后端的简单扩展
- Chat、RAG、Tool、Workflow、Governance 在系统中的位置
- 为什么课程必须围绕同一条业务主线展开

### 课堂重点

- 从请求输入变化开始讲，而不是从技术名词开始讲
- 先画出系统地图，再进入后续课程

### 课堂收束语

- 这套课教的是一种新的后端系统组织方式，而不只是模型 SDK 用法

## 第1课 Spring Boot 接入模型

### 开场问题

- 为什么把模型 SDK 直接写进 Controller 会让系统很快失控

### 本课必须讲清楚

- HTTP、SDK、统一模型网关三种接入方式
- 为什么 Java / Spring Boot 特别适合做模型接入边界层
- 当前 demo 的模型网关为什么要先于具体供应商能力存在

### 课堂重点

- 接口边界比模型供应商更重要
- 当前仓库是最小接入骨架，不是完整供应商集成平台

### 课堂收束语

- 模型接入不是把 SDK 接上，而是把模型能力接成后端可维护边界

## 第2课 对话接口与流式输出

### 开场问题

- 为什么很多 AI Demo 可以聊天，却很难变成稳定接口

### 本课必须讲清楚

- 同步接口和流式接口分别解决什么问题
- 为什么两者要共享一条业务边界
- 为什么响应结构不能直接暴露模型供应商格式

### 课堂重点

- 聊天不是“模型返回文本”而是“后端输出协议”
- 流式接口的错误路径同样要设计

### 课堂收束语

- 真正可用的聊天能力，首先是接口能力，其次才是模型效果

## 第3课 Prompt、上下文与结构化输出

### 开场问题

- 为什么模型看起来会回答，但系统输出仍然不稳定

### 本课必须讲清楚

- Prompt 必须参数化
- 上下文应该成为独立能力层
- 结构化输出为什么是业务可接入的关键

### 课堂重点

- 输出约束不是“优化提示词”，而是系统边界设计
- 上下文不能无限堆积，必须受控

### 课堂收束语

- AI 后端真正需要的不是更花哨的 Prompt，而是可解释、可解析、可接入的结果

## 第4课 文件解析与知识入库

### 开场问题

- 为什么文档上传成功不等于知识已经可用

### 本课必须讲清楚

- 文档从文件到知识片段要经过哪些处理步骤
- 为什么切片与 overlap 是 RAG 前的关键步骤
- 为什么片段必须保留元信息

### 课堂重点

- 这一课讲的是知识进入系统的入口，不是“上传组件”本身
- 教学上优先把链路跑通，再谈复杂基础设施

### 课堂收束语

- 文件上传只是开始，知识可检索才是真正进入系统

## 第5课 RAG 检索增强

### 开场问题

- 为什么系统已经有文档片段，回答仍然可能不可信

### 本课必须讲清楚

- 最小 RAG 链路由哪些步骤组成
- 为什么 citations 是必须字段
- 为什么检索为空时要显式降级

### 课堂重点

- RAG 的关键不是“查一下再答”，而是“让回答有依据”
- 不知道就显式说不知道，是系统能力，不是能力不足

### 课堂收束语

- RAG 的价值在于让 AI 回答从“像知道”变成“有依据”

## 第6课 Redis、ES、向量检索协同

### 开场问题

- 为什么只上一个向量库并不能解决全部检索问题

### 本课必须讲清楚

- Redis、关键词检索、向量召回各自负责什么
- 为什么要有混合检索边界
- 为什么当前 demo 先用最小抽象而不直接上完整基础设施

### 课堂重点

- 三类底座不是替代关系，而是协作关系
- 先讲职责，再讲产品选型

### 课堂收束语

- 检索效果来自职责协同，不来自单一底座神化

## 第7课 Agent 与 Tools

### 开场问题

- 为什么系统一旦能调用工具，风险等级会立刻上升

### 本课必须讲清楚

- Tool 和 Agent 不是同一个概念
- 工具为什么必须先注册、再开放
- allowlist 为什么是 Tool Calling 的底线

### 课堂重点

- 先把工具系统设计清楚，再讨论智能代理
- 这节课讲的是边界，不是“让 Agent 更万能”

### 课堂收束语

- Tool Calling 的关键不是功能多，而是权限清楚、可审计、可控

## 第8课 Workflow 与 MQ

### 开场问题

- 为什么知识入库、文档解析这类链路不应该继续写成同步接口

### 本课必须讲清楚

- Agent 与 Workflow 的边界
- Workflow 与 MQ 分别解决什么问题
- 为什么失败步骤必须结构化暴露出来

### 课堂重点

- 固定流程交给 Workflow，不要全部甩给 Agent
- 异步任务的本质是解耦和可观察，不是“为了用 MQ”

### 课堂收束语

- 长链路任务的价值不在于异步本身，而在于流程清晰、状态可见

## 第9课 AI 工程化治理

### 开场问题

- 如果系统今天开始输出高风险结果，你怎么第一时间知道

### 本课必须讲清楚

- 为什么 AI 系统比普通后端更需要治理
- SafetyPolicy、请求日志、指标门面、evals、CI 组成怎样的最小闭环
- 为什么治理不是收尾工作

### 课堂重点

- 风险边界必须落到后端规则中
- 日志、指标、评测各自职责不同
- 这一课应与前面的 Tool、RAG、数据助手形成联动

### 课堂收束语

- 没有治理的 AI 系统，不是上线慢，而是根本不具备上线资格

## 第10课 业务融合与数据助手

### 开场问题

- AI 能进入业务系统，但为什么不能直接接管业务执行

### 本课必须讲清楚

- 为什么选只读 SQL 助手做业务融合案例
- 为什么默认只返回预览而不直接执行
- 为什么结构化返回对业务接入比“能生成 SQL”更重要

### 课堂重点

- 业务融合先做增强层，不做执行层
- 数据助手的接口设计比模型效果更值得讲

### 课堂收束语

- AI 真正进入业务的前提，不是更大胆，而是更克制

## 第11课 全链路整合与生产化

### 开场问题

- 现在这套 demo 离真实生产到底还差什么

### 本课必须讲清楚

- 整套课程主链路如何收束成一个完整故事
- 从 demo 到生产应该优先补哪几层
- 为什么生产化是渐进替换，不是推倒重来

### 课堂重点

- 把 12 课内容重新压成三条主线复盘
- 让学员回答“主链路、边界、迁移路径”三个问题

### 课堂收束语

- AI-Native Java 后端不是一套新框架，而是一套新的系统组织与交付方法

## 结课收束建议

课程最后建议让学员完成两件事：

1. 画出自己的 AI 后端系统地图
2. 写出一份从当前 demo 到真实业务场景的迁移路径

如果学员已经能把这两件事做清楚，说明这套课真正完成了从“了解概念”到“具备工程落地意识”的转换。
