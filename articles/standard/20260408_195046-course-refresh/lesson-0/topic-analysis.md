# 第0课 话题分析

## 核心概念

这一课不是讲某个 AI 工具怎么点按钮，而是建立整套训练营的工作流总图。真正要解决的问题是：为什么团队里已经有人用 AI 了，但整体交付速度、返工率和协作质量并没有同步改善。

## 目标受众与深度

- 目标受众：Java / Spring Boot 后端工程师、技术负责人、需要设计 AI 协作流程的团队带头人
- 受众深度：中级为主，兼顾有落地压力的高级工程师
- 阅读目标：建立“AI 不是替代整个研发流程，而是嵌入每个研发动作”的认知

## 需要展开的深度杠杆

- 机制：个人提效为什么不会自动转成团队提效
- 取舍：AI 适合做候选方案生成，不适合直接拍板哪些结论
- 失败模式：无上下文、无验证、无沉淀时最容易出现什么问题
- 实施细节：如何把需求、设计、编码、测试和发布串成闭环
- 示例：结合仓库中的课程设计文档、`demo/` 和测试文件说明“真实上下文”长什么样
- 决策清单：什么时候该用 AI，什么时候应该先补上下文和验证方式

## 成功标准

- 学员能说清 AI 在后端研发流程中的正确位置
- 学员能画出至少一条“输入 -> AI 输出 -> 人工判断 -> 验证 -> 模板沉淀”的闭环
- 学员知道哪些工作可以交给 AI 做候选，哪些决定必须由工程师拍板
- 学员读完后能为自己团队列出两三个最值得先试点的动作
